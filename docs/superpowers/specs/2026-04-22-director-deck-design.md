# Director Deck — Design Spec

**Date:** 2026-04-22  
**Status:** Approved for implementation planning

---

## Overview

Director Deck is an Amplifier bundle (`amplifier-bundle-director-deck`) that takes a plain
text prompt and produces a polished PowerPoint deck with AI-generated images, backdrops, and
Veo 3.1 video transitions between slides. It is modelled on the "superpowers for decks"
philosophy: humans are involved at every aesthetic and content checkpoint, and expensive
generation only happens after human approval.

The workflow mirrors the Dan Shapiro DirectorDeck process:

> prompt → ghost deck → wireframe PPTX → images & backdrops → transitions → final PPTX

---

## Architecture

### Artifact Type

A single Amplifier bundle with:
- 5 specialised agents
- 4 Python utility tools
- 1 staged recipe with 3 human approval gates

### Final Deliverable

A `.pptx` file with:
- Full slide content and design
- Per-slide AI-generated content images and backdrop images
- Veo 3.1 MP4 transition clips embedded natively between slides (autoplay on slide advance)

### Intermediate Deliverables (per run)

| File | Purpose |
|---|---|
| `DESIGN.md` | Visual identity for the deck (Stitch DESIGN.md spec format) |
| `slide_deck.json` | Structured slide content — the shared contract between all agents |
| `deck_wireframe.html` | Browser-viewable wireframe with placeholder image areas |
| `deck_wireframe.pptx` | PowerPoint wireframe for review in PowerPoint/Keynote |
| `deck_enriched.html` | HTML with real generated images in place |
| `deck_enriched.pptx` | PPTX with real images |
| `keyframes/slide-N.png` | Playwright screenshot of each enriched HTML slide |
| `transitions/slide-N-to-M.mp4` | Veo transition clip per slide pair |
| `final_deck.pptx` | Final assembled PPTX with embedded MP4 transitions |

All runs are stored under `./runs/<date>-<slug>/` so previous runs are never overwritten.

## Shared Data Contract: `slide_deck.json`

All agents read and progressively enrich a single `slide_deck.json` file. It is the
single source of truth for slide content and asset references.

```json
{
  "meta": {
    "title": "Series A Pitch",
    "prompt": "...",
    "slide_count": 10
  },
  "slides": [
    {
      "id": 1,
      "title": "The Problem",
      "bullets": ["Point A", "Point B"],
      "speaker_notes": "Emphasise the scale of the pain.",
      "image_brief": "A lone developer staring at a wall of error logs, cinematic",
      "backdrop_brief": "Dark gradient, subtle circuit board texture, no text",
      "assets": {
        "image": "assets/slide-1-image.png",
        "backdrop": "assets/slide-1-backdrop.png"
      },
      "transition_to_next": "transitions/slide-1-to-2.mp4"
    }
  ]
}
```

Agents fill in `assets` (visual-director) and `transition_to_next` (transition-director)
progressively. The deck-stitcher reads the fully populated file to assemble the final PPTX.

## Visual Identity Contract: `DESIGN.md`

`DESIGN.md` follows the [Google Stitch DESIGN.md specification](https://stitch.withgoogle.com/docs/design-md/specification/).
It is generated by `ghost-deck-writer` from the initial prompt and acts as the visual
design system for the entire deck.

It combines YAML frontmatter (machine-readable design tokens) with markdown prose (human-readable
design rationale). Every downstream agent reads it:

| Agent | Uses DESIGN.md to... |
|---|---|
| `slide-architect` | Map color/typography/spacing tokens to CSS |
| `visual-director` | Craft image prompts that match palette and mood |
| `transition-director` | Pass style context to Veo prompts |
| `deck-stitcher` | Apply theme to PPTX slide master |

Example `DESIGN.md` for a tech pitch deck:

```yaml
---
name: Series A Pitch
colors:
  primary: "#0F172A"
  accent: "#38BDF8"
  surface: "#1E293B"
  on-surface: "#F1F5F9"
typography:
  h1:
    fontFamily: Inter
    fontSize: 48px
    fontWeight: 700
  body-md:
    fontFamily: Inter
    fontSize: 18px
    fontWeight: 400
spacing:
  slide-padding: 64px
  section-gap: 32px
---

## Overview
High-stakes tech pitch. Clean, confident, dark-mode-first. Every slide should
feel like a Bloomberg terminal meets a well-funded startup.

## Colors
- **Primary (#0F172A):** Near-black slide backgrounds.
- **Accent (#38BDF8):** Sky blue for headlines and key callouts only.
- **Surface (#1E293B):** Card and panel backgrounds.

## Do's and Don'ts
- Do use accent only on the single most important element per slide
- Don't use more than 2 fonts
```

The recipe validates `DESIGN.md` with `npx @google/design.md lint DESIGN.md` after
generation. Broken token references or WCAG contrast failures block Gate 1 from opening.

## Agents

### `ghost-deck-writer`
**Input:** User prompt  
**Output:** `slide_deck.json` + `DESIGN.md`

Produces the full slide structure (titles, bullets, speaker notes, image briefs, backdrop
briefs) and infers an appropriate visual identity from the topic and tone. The DESIGN.md
prose sections are written to reflect the emotional register of the content.

### `slide-architect`
**Input:** `slide_deck.json` + `DESIGN.md` + optional `assets/`  
**Output:** `deck_wireframe.html` + `deck_wireframe.pptx` (first pass) or `deck_enriched.html` + `deck_enriched.pptx` (second pass)

Generates HTML slides as `960×540px` divs, with DESIGN.md tokens injected as CSS variables.
Pre-Gate 1: image areas are styled placeholder divs with brief descriptions overlaid.
Post-Gate 2: `<img>` tags pointing to real generated assets.

Also generates a `.pptx` via `pptx_builder.py` with the same layout.

### `visual-director`
**Input:** `slide_deck.json` (image briefs) + `DESIGN.md`  
**Output:** `assets/slide-N-image.png`, `assets/slide-N-backdrop.png`

Uses `amplifier-module-tool-openai-images` (singh2) to generate:
- Per-slide content images (from `image_brief`)
- Per-slide backdrop images (from `backdrop_brief`)

DESIGN.md color palette and mood prose are woven into each prompt for consistency.

### `transition-director`
**Input:** `deck_enriched.html` (enriched HTML slides)  
**Output:** `keyframes/slide-N.png`, `transitions/slide-N-to-M.mp4`

1. Uses `screenshot_tool.py` (Playwright headless Chromium) to capture each slide at `960×540px`
2. Crafts a Veo prompt per slide pair using DESIGN.md `Overview` prose for style context
3. Calls Veo 3.1 in `image_to_video` interpolation mode — slide N as first frame, slide N+1 as last frame
4. Produces 4-second MP4 clips per slide pair

Veo prompt pattern:
```
Smooth cinematic transition. <DESIGN.md Overview prose>. No text, no people.
Motion flows naturally between the two keyframes.
```

### `deck-stitcher`
**Input:** `slide_deck.json` + `assets/` + `transitions/` + `DESIGN.md`  
**Output:** `final_deck.pptx`

Assembles the final PPTX via `pptx_stitcher.py`:
- Builds fully styled slides with real images using DESIGN.md tokens on the slide master
- Injects MP4 transition clips into the PPTX zip package as `<p:transition>` XML elements
- Transition clips autoplay on slide advance (Office 2013+ compatible)

## Recipe: `director-deck.yaml`

### Stage 1 — Content & Identity

1. `ghost-deck-writer` → `slide_deck.json` + `DESIGN.md`
2. `npx @google/design.md lint DESIGN.md` (auto quality gate — blocks on error)
3. `slide-architect` → `deck_wireframe.html` + `deck_wireframe.pptx`

**Gate 1 — Content & Identity Review**
- Human opens `deck_wireframe.html` in browser and `deck_wireframe.pptx` in PowerPoint
- Human may edit `slide_deck.json` (content) and/or `DESIGN.md` (visual identity) directly
- Approval message may carry instructions (e.g. "make slide 3 punchier, change accent to teal")
- Recipe re-runs Stage 1 agents as needed before accepting approval

### Stage 2 — Visual Generation

1. `visual-director` → `assets/slide-N-image.png` + `assets/slide-N-backdrop.png`
2. `slide-architect` (second pass) → `deck_enriched.html` + `deck_enriched.pptx`

**Gate 2 — Visual Review**
- Human opens `deck_enriched.html` in browser
- Approval message may request specific re-generations (e.g. "redo slide 5 backdrop")
- `visual-director` re-runs for specified slides only; rest of assets preserved

### Stage 3 — Transitions

1. `transition-director` → `keyframes/slide-N.png` + `transitions/slide-N-to-M.mp4`

**Gate 3 — Transition Review**
- Human previews transition MP4 clips
- Approval message may request specific clip regenerations
- `transition-director` re-runs for specified pairs only

### Stage 4 — Final Assembly

1. `deck-stitcher` → `final_deck.pptx`

No gate — assembly is deterministic from approved assets.

## Technical Implementation Details

### HTML Slides (`html_renderer.py`)

Each slide is a `960×540px` div. DESIGN.md YAML frontmatter is parsed and injected as
CSS custom properties:

```css
:root {
  --color-primary: #0F172A;
  --color-accent: #38BDF8;
  --font-h1-size: 48px;
  --spacing-slide-padding: 64px;
}
```

The `slide-architect` agent generates the per-slide HTML layout from slide content,
driven by these tokens. No static templates — layouts are generated per-slide.

### PPTX Transition Embedding (`pptx_stitcher.py`)

PowerPoint embeds MP4 transitions via the `<p:transition>` XML element in each slide's
XML, with the video file copied into `ppt/media/` inside the PPTX zip package.
`python-pptx` is used with low-level XML manipulation for the transition injection.

Compatible with Office 2013+ on macOS and Windows.

### Playwright Screenshots (`screenshot_tool.py`)

Launches headless Chromium, loads each slide's HTML at `960×540px` viewport, captures
a PNG. These are the Veo keyframes. The `960×540` size ensures correct 16:9 aspect ratio
for both PPTX slides and Veo input.

### Veo Transition Generation

Uses Veo 3.1 `image_to_video` with interpolation (first frame + last frame):

```python
veo(
    operation="image_to_video",
    image_path="keyframes/slide-N.png",
    last_frame_path="keyframes/slide-N+1.png",
    prompt="...",
    duration_seconds="4",
    model="veo-3.1-generate-preview"
)
```

### DESIGN.md Validation

```bash
npx @google/design.md lint DESIGN.md
```

Run automatically after `ghost-deck-writer`. Exit code 1 (broken refs, contrast failures)
blocks the pipeline before any visual generation begins.

## What Is Not In Scope

- Animations within a single slide (bullets flying in, etc.) — out of scope for v1
- Custom slide templates / importing existing decks — out of scope for v1
- A web UI — CLI / recipe invocation only for v1
- Non-PPTX output formats (PDF, reveal.js, etc.) — out of scope for v1
